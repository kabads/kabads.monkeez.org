1.5 Device nodes

1.5.1 Character and block devices have filesystem entries associated with them; network devices in Linux do not

1.5.2 These device nodes can be used by programs to communicate with devices, using normal I/O system calls such as open(), close(), read() and write().

1.5.3 On the other hand, network devices work by transmitting and receiving prackets, which must be constructed by breaking up streams of data, or reassembled in to streams when recweived.

1.5.4 a device driver may manage multiple device nodes, which are normally placed in the /dev directory

1.5.5 device nodes can be created with sudo mknod [-m mode] /dev/name <type> <major> <minor>

1.5.6 e.g. mknod -m 666 /dev/mycdrv c 254 1

1.6 Major and minor numbers

1.6.1 the major and minor numbers identify the driver associated with the device, with the driver uniquely reserving a group of numbers.

1.6.2 In most cases (but not all) device nodes of the same type (block or character) with the same major number use the same driver

1.6.3 If you list some device nodes as in ls -l /dev/sda* the major and minor numbers appear in the same place that the file size would when looking at a normal file, in the example as 8, 1 etc. Normal users will probably never use this, but sometimes sysadmins need to unpick these if the system gets confused about devices, or has some hardware at run time.

1.6.4 The minor number is used by the device driver to differentiate between the different devices it may control, or how they are used. This might be different instances of the same kind of device (e.g. sound cards or hdd partition) or different modes of operation for a given device (such as a different density floppy drive media).

1.6.5 Device numbers have meaning in user-space as well. Two system calls mknod() and stat() return information about major and minor numbers.

1.6.6 Problems with kernel 2.4

  1. Lots of device nodes were created with kernel 2.4 just in case - this proved to be a problem - trimming this was a manual job which was error-prone
  2. having huge directories filled with these slowed access times to device nodes
  3. running out of major and minor numbers required a more modern and dynamic approach - numbers cannot be got rid of as the POSIX standard requires them.

1.7 udev method creates device nodes on the fly as they are needed.

1.7.1 u in udev stands for user indicating that most of the work of creating, removing, and modifying devices nodes is done in user-space

1.7.2 udev handles the dynamical generation of device nodes and it replaced devfs and hotplug.

1.7.3 One nice feature is persistent device naming - names need not depend on the order of device connection or plugging in. Such behaviour is controlled by udev rules

1.7.4 udev rules

  1. runs as a daemon - either udevd or systemd-udevd monitoring a netlink socket
  2. when a new device is plugged in the uevent kernel facility sends a message through the socket, which udev receives and takes appropriate action to create or remove device nodes of the right names and properties according to the rules.
  3. the three components of udev are:
    1. libudev library which allows access to information about the devices
    2. the udevd daemon which manages the dev directory
    3. the udevadm utility for control and diagnostics
  4. As devices are added or removed, working with the hotplug subsystem, udev:
    1. acts upon a notification of events to create and remove device nodes
    2. The info necessary to create them with right names, major and minor numbers, permissions, etc are gathered by examination of information already registered in the sysfs pseudo-filesystem (mounted at /sys) and a set of config files.
    3. The main config file is /etc/udev/udev.conf - it contains:
      1. where to place device nodes
      2. default permissions and ownership
    4. By default, rules are located in /etc/udev/rules.d directory
    5. By reading the man page for udev one can get a lot of specific information about how to set up rules for common situations
    6. When udev receives a message from the kernel about a device being added or removed, it parses the rule-setting files in /etc/rules.d/*.rules
    7. It then takes appropriate actions, including:
      1. device node naming
      2. device node and symbolic links creation
      3. setting file permissions and ownership for the device node
      4. taking other actions to initialize and make the device available
    8. udev rules files are located under /etc/udev/rules.d/<rulename>.rules with names like:
      1. 30-usb.rules
      2. 90-mycustom.rules
    9. udev reads the rule files ending with a suffix of .rules
    10. If it finds more than one, it reads them both one by one, lexicographically i.e. in ascending alphabetical order.
    11. The standard rule file name is generally a two digit number followed by a descriptive name (for the rules), followed by the .rules suffix
    12. The format for a udev fule is simple:
      1. <match><op>value [, …] <assignment><op>value[, …]
      2. There are two separated parts defined on a single line
        1. The first part consists of one or more match pairs denoted by ==. These try to match a device's attributes and/or characteristics to some value
        2. The second part consists of one or more assignment key-vaule pairs that assign a value to a name, such as a file name, group assignment, even file permissions etc.
      3. If no matching rule is found, it uses the default device node name and other attributes
    13. Here's an example of a rules file for a Fitbit device:
      1. cat /etc/udev/conf.d/rules.d/99-fitbit.rules:
      2. SUBSYSTEM=="usb", ATTR{idVenor}=="2687", ATTR{idProduct}=="fb01", SYMLINK+="fitbit", MODE="0666"
