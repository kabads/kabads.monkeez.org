1.4 Filesystem Hierachy Standard administered by the Free Standards Group and now the Linux Foundation

1.4.1 Specifies the main directories that need to be present, and describes their purposes. http://www.pathname.com/hfs

1.4.2 Typical layout

Location FHS? Description
/bin Yes Essential executable programs that must be available in single user mode.
/boot Yes Files needed to boot the system, such as the kernel, initrd or initramfs images, and boot configuration files and bootloader programs.
/dev Yes Device Nodes, used to interact with hardware and software devices.
/etc Yes System wide configuration files.
/home Yes User home directories including personal settings, files, etc.
/lib Yes Libraries required by executable binaries in /bin and /sbin.
/lib64 No 64-bit libraries required by executable binaries in /bin and /sbin, for systems which can run both 32-bit and 64-bit programs.
/media Yes Mount points for removable media such as CDs, DVDs, USB sticks etc.
/mnt Yes Temporarily mounted filesystems.
/opt Yes Optional application software packages.
/proc Yes Virtual pseudo-filesystem giving information about the system and processes running on it. Can be used to alter system parameters.
/sys No Virtual pseudo-filesystem giving information about the system and processes running on it. Can be used to alter system parameters. Similar to a device tree and is part of the Unified Device Model.
/root Yes Home directory for the root user.
/sbin Yes Essential system binaries.
/srv Yes Site-specific data served up by the system. Seldom used.
/tmp Yes Temporary files; on many distributions lost across a reboot and may be a ramdisk in memory.
/usr Yes Multi-user applications, utilities and data; theoretically read-only.
/var Yes Variable data that changes during system operation.

1.4.3 /

  1. boot the system
  2. Restore the system from system backups on external media or NAS etc.
  3. Recover and/or repair the system; an experienced maintainer must have the tools to diagnose and reconstruct a damaged system
  4. According to FHS no application or package should create new subdirectories of the root directory

1.4.4 /bin - all single user commands

  1. contains executable programs and scripts needed by both sysadmins and unprivileged users, which are required when no other filesystems have yet been mounted, for example, when booting in to single user or recovery mode
  2. may also contain executables which are used indirectly by scripts
  3. may not include any subdirectories
  4. command binaries deemed non-essential enough for /bin go in /usr/bin - programs for non-root users are placed in this category

1.4.5 /boot - all the files for booting the system

  1. vmlinuz -kernel compressed
  2. initramfs - initial ram filesystem - mounted before the real root filesystem becomes available.
    1. sometimes initramfs is called initrd (initial ram disk) which was the older method, but the name survives.
  3. /boot/config is here for compiling the kernel- just here for bookeeping and reference for debugging
  4. System.map - the kernel symbol table, which is useful for debugging. It gives the hexadecimal addresses of all kernel symbols

1.4.6 /dev

  1. special device files (or device nodes) which represent devices built in to the connected system.
  2. such devices represent character (byte-stream) and block I/O devices
  3. network devices do not have a device node in linux but instead refer to eth1 or wlan0
  4. all latest linux vesions use udev system which creates a node in /dev only as needed when devices are found at boot or when plugged in.
  5. an unmounted filesystem would have dev as empty - things are populated as udev requires
  6. embedded linux systems may have fixed device nodes instead of /dev and not use udev but mdev

1.4.7 /etc - use machine-local config files and some startup scripts

  1. no executables allowed
  2. distros add configuration files and directories to /etc
  3. RHEL adds /etc/sysconfig where system config files live
  4. /etc/skel which contains skeleton files used to populate new newly created home directories and /skel/etc/init.d which contains startup and shutdown scripts when using System V initialization

1.4.8 /home - user directories

  1. All personal configuration, data and executable programs are placed in this directory
  2. /home may also contain subdirectories for various groups - e.g. /home/students
  3. users can substitute the environmental variable $HOME for their root directory
  4. one exception is /root is the home for the root user

1.4.9 /lib - libraries needed to execute binaries in /bin and /sbin for booting the system

  1. kernel modules are held under /lib/modules/<kernel-version>
  2. PAM (pluggable authentication modules) are stored in /lib/security
  3. systems which support both 32-bit and 64-bit binaries must keep both kinds of libraries on the systtem.
    1. /lib64 and /lib can co-exist

1.4.10 /media is typically used to mount filesystems or removable media - cds usbs dvds

  1. udev can dynamically create directories under /media and then mount the removable filesystem there with names for udev rules (speficied in config files).
  2. udev will also remove directories under /media as things are umounted.
  3. if the media has more than one partition, then more than one entry will appear on /media
  4. on newer distributions - SUSE/RHEL 7, media will pop up under /run/media/username

1.4.11 /mnt - for networked file systems

  1. NFS
  2. Samba
  3. CIFS
  4. AFS
  5. Historically, /mnt was used to hold these types of files - but not now in modern systems

1.4.12 /opt - software packages that wish to keep all of their files in one isolated place rather than scatter them all over the filesystrem.

  1. /opt/dolphyapp would be the directory where all the files exist for dolphyapp
  2. this makes it relatively easy to install an app
  3. However, if you are using a package management, it is easy to install and uninstall with a clear sense of file manifests and locations, wihtout exhibiting such anti-social behaviour.
  4. /opt is used by proprietary vendors or those who like to avoid distribution complexities
  5. Directories /opt/bin /opt/doc /opt/include /opt/info, /opt/lib and /opt/man are reserved for the local sysadmin use. Packages may provide files which are linked or copied to these directories, but the packages must also be able to function without the programs being in these special directories.

1.4.13 /proc - pseudo-filesystem (held in memory and not on disk) - empty on a non-running system

  1. Kernel exposes some important data structures through /proc entries. Each active process on the system has its own subdirectory that gives detailed information about the process, its resources and its history.
  2. Entries in /proc are termed virtual files - they are listed as zero bytes in size, but they can contain a lot of information.
  3. Most of the time and dates settings are held in virtual files and are being constantly updated. They are obtained when the file is requested.
  4. Important pseudo-files are /proc/interrupts, /proc/meminfo, /proc/mounts and /proc/partitions provide information which is up-to-date about the system hardware.
  5. files containing information on a similar topic are grouped in to virtual directories.

1.4.14 /sys - sysfs pseudo-filesystem - in memory - empty on non-running system

  1. it is younger than /proc and adheres to stricter standards about what kind of entries can exist here.

1.4.15 /root ("slash-root") is the home directory for the root user- superuser privilege is required.

1.4.16 /sbin contains binaries for booting, restoring, recovering and/or repairing in addition to those binaries in /bin.

  1. These binaries must be able to mount other filesystems on usr and /home and other locations if needed.
  2. Some distributions are abandoning the idea of having /sbin and /usr/sbin and having symlinks to preserve a two directory view.

1.4.17 /tmp stores temporary files and can be accessed by any user or application. They do not last for a long time.

  1. Some distributions have a cron job that will delete files older than 10 days old. RHEL 6.
  2. Some distributions remove contents of /tmp for every boot. Ubuntu
  3. Some distributions utilize a virtual filesystem using the /tmp directory only as a mount point for ram disk using tmpfs filesystem. Fedora.
    1. In this case, one must avoid large files in /tmp as it will actually occupy memory instead of the disk and it is easy to harm or crash the system through memory exhaustion.
    2. This can be cancelled using systemd by doing the following: systemctl mask tmp.mount followed by a reboot.

1.4.18 /usr can be thought of as a secondary hierarchy. It holds files which are not needed for system booting.

  1. Software packages should not create subdirectories under /usr
  2. Some symbolic links may exist to other locations for compatiability purposes.
  3. This is typically read-only data containing binaries which are not needed in single user mode.
  4. usr/local directory where local binaries and such may be stored
  5. /usr/share/man - man pages can be stored here.
    Directory Purpose
    /usr/bin Non-essential binaries and scripts, not needed for single user mode. Generally, this means user applications not needed to start the system.
    /usr/include Header files used to compile applications.
    /usr/lib Libraries for programs in /usr/bin and /user/sbin.
    /usr/lib64 64-bit libraries for 64-bit programs in /usr/bin and /usr/sbin.
    /usr/sbin Non-essential system binaries, such as system daemons.
    /usr/share Shared data used by applications, generally architecture-independent.
    /usr/src Source code, usually for the Linux kernel.
    /usr/X11R6 X Window files; generally obsolete. Not present in many modern Linux distributions.
    /usr/local Local data and programs specific to the host. Subdirectories include bin, sbin, lib, share, include, etc.

1.4.19 /var variable data files that change frequently during system operation

  1. Log files
  2. Spool files
  3. Administrative data files
  4. Transient and temporary files, such as cache contents
  5. Cannot be mounted as read-only filesystem
  6. If it fills up it should not lock up the system
  7. /var/log is where most of the log files are rooted
  8. /var/spool is where processes such as mail, printing and cron jobs are stored while waiting action
  9. /var/ftp used for server base
  10. /var/lib persistent data modified by programs as they run
  11. /var/lock lock files used to control simultaneous access to resources
  12. /var/mail user mailboxes
  13. /var/run information about the running system since last boot
  14. /var/tmp temporary files to be preserved across system reboot, sometimes linking to /tmp
  15. /var/www root for website hierarchies
